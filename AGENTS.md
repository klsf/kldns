# AGENTS.md

本文件用于约束 AI 在本仓库内编写、重构和维护代码的行为。所有代码生成、修改、重构、测试和文档更新都必须优先遵守这里的规则。

## 项目目标

KLDNS 是面向二级域名分发场景的 DNS 管理系统，核心能力包括：

- 多用户、多主域、多 DNS 平台统一管理。
- 用户自助申请前缀、维护解析记录。
- 主域开放策略、用户组、积分、备案状态和说明文案管理。
- 用户花费积分注册二级域名，注册后管理其下任意支持的解析记录。
- 开放 API、API Token 管理和域名解析操作。
- 操作日志、异常检测、记录清理和后台运维管理。

本次重构目标技术栈：

- 后端：Golang + Beego + SQLite3。
- 前端：Vite + Vue 3 + Element Plus。
- 数据库：SQLite3，必须配套可追踪的迁移脚本。

## 总体原则

- 先理解现有业务，再写代码；不要只按文件名猜测逻辑。
- 保持代码可读、可维护、可测试，禁止把所有逻辑堆在一个文件中。
- 每个文件只承担清晰的职责，函数保持短小，复杂流程拆分为命名明确的私有函数或服务方法。
- 遵循项目既有目录、命名和风格；没有既有模式时，按本文件约定建立结构。
- 修改范围要尽量小，不做与任务无关的大规模重构。
- 不要删除用户已有改动，不要回滚不属于当前任务的文件。
- 任何涉及认证、权限、积分、DNS 真实接口、二级域名归属、数据迁移的改动，都要格外保守，并补充验证说明。

## 后端编码规则

- `main.go` 只负责启动流程和基础装配，不写业务逻辑。
- Controller 只做参数解析、鉴权上下文读取、调用 service、返回响应。
- Service 承载业务规则，例如二级域名注册扣费、DNS 同步、权限校验、日志记录。
- Repository 只处理数据读写，不夹带 HTTP、DNS API 或页面逻辑。
- DNS 平台必须通过统一接口接入，例如 `Provider`、`RecordManager`、`ZoneManager`；新增平台只能增加适配器，不要修改业务流程代码。
- 数据库访问必须使用参数化查询或 ORM 安全能力，禁止拼接用户输入形成 SQL。
- SQLite 写入要考虑事务；同一个业务动作涉及多表时必须放在同一个事务中。
- 统一错误返回格式，业务错误要有稳定错误码，不要到处返回裸字符串。
- 所有时间使用明确时区策略；数据库建议存 UTC 或 Unix 时间戳，展示层再转换。
- 密码、Token、DNS 密钥必须加密或哈希存储；日志中禁止输出明文密钥、Token、密码。
- API 权限检查必须靠后端完成，前端隐藏按钮不能替代后端授权。

## Beego 约束

- 保持 Beego 推荐的根目录结构，不要新增 `backend/` 包裹后端代码，也不要使用 `internal/` 重建一套目录。
- 路由注册集中放在 `routers`，不要散落在控制器文件里。
- 控制器按资源拆分，例如 `DomainController`、`SubdomainController`、`RecordController`、`TokenController`、`DnsProviderController`。
- 请求结构体、响应结构体放在 `dto`，不要在控制器里临时拼大 map。
- 中间件放在 `middlewares`，负责认证、恢复、请求日志和跨域，不要在每个控制器里重复写。
- Beego 框架对象不要向业务层扩散，service 和 repository 应尽量保持普通 Go 代码，便于测试。

## SQLite 与迁移

- 所有表结构变化都必须有迁移文件，不允许只改模型不改迁移。
- 迁移文件命名建议：`YYYYMMDDHHMMSS_description.sql`。
- 迁移必须可重复审查，包含清晰的 `up` 逻辑；如项目支持回滚，也要提供 `down` 逻辑。
- 关键表建议包括：
  - `users`
  - `groups`
  - `domains`
  - `dns_providers`
  - `records`
  - `subdomains`
  - `api_tokens`
  - `operation_logs`
  - `settings`
- 对唯一约束、外键、索引要显式声明；常用查询字段必须建立索引。
- 不要把 JSON 当成默认建模方式；只有平台差异化配置、扩展元数据等确实不稳定的字段才使用 JSON。

## 前端编码规则

- 使用 Vue 3 Composition API 和 TypeScript。
- 页面组件只负责编排页面，不要把 API、状态、校验、格式化全部写在一个 `.vue` 文件里。
- API 调用集中放在 `web/src/api`，按资源拆分，例如 `domains.ts`、`subdomains.ts`、`records.ts`。
- 复杂业务页面按 `features` 拆组件、hooks 和类型；单个 `.vue` 文件过长时必须拆分。
- Element Plus 表格、表单、弹窗和分页要封装成清晰组件，避免复制粘贴大段模板。
- 所有前端页面必须同时适配移动端和 PC 端；响应式布局是交付要求，不是后续优化项。
- 会员中心和管理后台在 PC 端必须采用左侧菜单、右侧内容区的后台式布局；左侧菜单承载主导航，右侧内容区承载当前页面。
- 页面标题区不要使用标题上方的英文 eyebrow/kicker 小字标签，例如 `PROVIDERS`、`OPERATIONS`、`Records`、`KLDNS` 这类装饰性英文小标题；页面主标题直接使用中文业务名称，必要说明放在标题下方的中文说明文本中。
- 页面布局优先使用弹性布局、栅格、断点和容器宽度约束，禁止依赖固定大宽度导致移动端横向溢出。
- 表格、筛选器、表单、弹窗、侧边栏等复杂组件必须设计移动端形态，例如折叠筛选、抽屉、卡片列表、横向滚动表格或精简列展示。
- 前端所有权限判断只用于展示优化，真实权限以后端返回和校验为准。
- 表单校验前后端都要做；前端负责体验，后端负责安全。
- 路由页面命名要直观，例如 `AdminDomainList.vue`、`AdminSubdomainList.vue`、`UserRecordList.vue`。
- `web/src/styles` 只放公共样式、主题变量、基础布局和 Element Plus 覆盖，不要放具体业务组件的私有样式。
- 组件自己的样式必须写在对应 `.vue` 文件的 `<style scoped>` 中；只有多个组件确实复用的样式才能提升到公共样式目录。
- 不要在组件内硬编码大量魔法字符串；状态枚举、解析类型、域名状态应放到常量或类型定义中。

## API 设计规则

- API 路径按资源设计，版本前缀建议使用 `/api/v1`。
- 请求和响应结构要稳定，不要把数据库模型直接原样暴露给前端。
- 列表接口必须支持分页，必要时支持搜索、筛选和排序。
- 修改类接口必须返回明确结果，失败时返回稳定错误码和人类可读消息。
- 涉及注册和解析的动作必须清楚区分：
  - 已直接生效。
  - 无权限。
  - 积分不足。
  - DNS 写入失败。
- 开放 API 必须使用 `Authorization: Bearer <token>`，并区分用户前台会话和 API Token 权限。

## 业务拆分建议

- `domains`：主域管理、开放策略、解析类型限制、备案状态。
- `records`：解析记录增删改查、格式校验、同步 DNS 平台。
- `subdomains`：二级域名注册、归属、状态和列表查询。
- `points`：积分扣减、退回、流水记录。
- `providers`：DNS 平台配置、凭据校验、连接测试。
- `tokens`：开放 API Token 创建、禁用、权限范围。
- `logs`：操作日志、审计查询。
- `settings`：系统配置、注册开关、清理策略。

不要把这些业务全部放进一个 controller、service、store 或页面文件。

## DNS 平台适配器规则

- 业务层只能依赖统一接口，不能直接调用某个 DNS 厂商 SDK。
- 每个平台适配器只处理该平台认证、请求签名、接口调用和响应转换。
- 适配器返回统一领域结构，例如 `Zone`、`Record`、`ProviderError`。
- 平台特有字段放在配置结构中，不要污染通用记录模型。
- 外部 API 错误要保留可排查信息，但不得泄露密钥。
- 真实 DNS 写操作必须有清晰日志，便于追溯是谁在什么时候修改了什么。

- 前端代码提交前至少运行：
  - `npm run typecheck`，如果项目提供。
  - `npm run lint`，如果项目提供。
  - `npm run build`。
- 涉及前端页面改动时，必须检查移动端和 PC 端显示效果，至少覆盖 375px、768px、1440px 三类宽度，确保内容不重叠、按钮可点击、表单可填写、弹窗不超出视口。

  - 解析记录格式校验。
  - 权限判断。
  - 积分扣减和退回。
  - 二级域名注册扣费和归属。
  - API Token 鉴权。
  - DNS provider 错误映射。

- 修改迁移后必须至少验证新库初始化流程。

## 代码质量红线

禁止出现以下做法：

- 把后端业务都写在 `main.go` 或单个 controller 文件里。
- 把前端页面、API 请求、状态管理、校验和样式全部塞进一个 `.vue` 文件里。
- 为了赶进度复制大段相似代码，不提取组件、函数或适配器。
- 绕过 service，直接在 controller 中写复杂 SQL 和业务判断。
- 在 repository 中调用 HTTP、DNS API 或读取 Web 请求上下文。
- 在前端硬编码后端地址、Token、管理员账号或 DNS 密钥。
- 打印或提交 `.env`、数据库文件、私钥、Token、AccessKey。
- 无迁移地修改数据库结构。
- 用全局变量保存用户态、请求态或可变业务状态。
- 靠前端隐藏按钮实现安全控制。

## 迁移旧项目时的要求

- 先梳理旧项目功能，再逐模块迁移，不要一次性重写所有业务。
- 保留业务语义，不要因为技术栈改变而丢掉二级域名注册、积分、日志、开放 API 等关键能力。
- 每迁移一个模块，都要建立后端接口、前端页面、基础测试和最小可运行验证。
- 新旧字段名可以调整，但必须维护清晰映射，避免数据迁移时丢失含义。
- 对旧项目中的 DNS 平台支持，要按适配器逐个迁移和验证。

## 文档要求

- 新增模块时同步更新 README 或相关开发文档。
- 配置项必须有示例，不要只在代码中隐式出现。
- API 变更要记录请求、响应和错误码。
- 迁移脚本要能让维护者理解为什么新增这些字段或索引。

## DNS 写操作一致性

DNS 真实写操作必须按可追溯、可补偿、可重试的方式设计。不要假设远端 DNS API 和本地 SQLite 事务可以组成一个原子事务。

涉及新增、修改、删除解析记录时，必须明确处理：

- 本地校验失败：不得调用远端 DNS。
- 远端 DNS 调用失败：不得写入本地成功状态，不得扣减积分。
- 远端 DNS 成功但本地写入失败：必须尝试补偿，例如删除刚创建的远端记录，或写入异常待修复任务。
- 二级域名注册、积分扣减、记录写入、操作日志必须在本地事务内保持一致。
- 重试不得造成重复添加记录或重复扣分。
- 每次远端 DNS 写操作必须记录操作者、来源、域名、主机记录、记录类型、记录值摘要、平台、远端 record id 和结果。
- 日志中的记录值可按业务需要展示，但密钥、Token、签名、Authorization 绝不能出现。

## SQLite 运行约束

SQLite 初始化时必须显式启用或确认：

- `PRAGMA foreign_keys = ON`
- 合理的 `busy_timeout`
- 根据部署模式评估是否启用 WAL，例如 `PRAGMA journal_mode = WAL`

Repository 层必须：

- 为写操作设置明确事务边界。
- 避免长事务中执行远端 DNS API、HTTP 请求或耗时任务。
- 对唯一约束冲突返回稳定业务错误码。
- 对数据库锁、超时、外键失败给出可诊断错误。
- 测试中使用临时数据库，并验证迁移后的外键、唯一约束和索引存在。
